## Observability

### Overview

This document describes the observability architecture implemented for the **Cloud Native GKE** project.

The project adopts a centralized observability strategy using **Prometheus**, **Grafana**, and **Alertmanager** to collect metrics, visualize platform health, and generate actionable alerts across the Kubernetes environment.

The observability stack monitors Kubernetes infrastructure, cluster components, platform services, and application workloads, providing comprehensive visibility into the health and performance of the environment.

Observability is a fundamental capability of the platform, enabling proactive monitoring, faster incident detection and troubleshooting, capacity planning, performance analysis, and data-driven operational decisions. All monitoring components are deployed and managed declaratively through GitOps, ensuring a consistent, repeatable, and production-inspired observability solution for workloads running on **Google Kubernetes Engine (GKE)**.

## Monitoring Stack

The platform uses **kube-prometheus-stack** as the monitoring solution.

Core components include:

| Component | Purpose |
|-----------|----------|
| Prometheus | Metrics collection |
| Grafana | Dashboards and visualization |
| Alertmanager | Alert routing |
| ServiceMonitor | Metrics discovery |
| PrometheusRule | Alert definitions |

---
## Metrics Collection

Metrics are collected from multiple sources.

### Kubernetes

- Nodes
- Pods
- Deployments
- StatefulSets
- Services
- Namespaces

---
### Applications

Application metrics include:

- HTTP requests
- Request latency
- Error rate
- Response time
- CPU usage
- Memory usage

Each application exposes Prometheus-compatible metrics.

---
### Databases

Database monitoring is provided using exporters.

Current exporters include:

| Exporter | Purpose |
|----------|----------|
| PostgreSQL Exporter | PostgreSQL metrics |
| Redis Exporter | Redis metrics |

Collected metrics include:

- Connections
- Memory usage
- Cache statistics
- Query activity
- Replication status

---
## Alerting

Prometheus evaluates alert rules continuously.

Alertmanager routes alerts to configured notification channels.

Typical alerts include:

- High CPU usage
- High memory utilization
- Pod restart loops
- Application downtime
- Database connectivity failures
- Kubernetes node failures

Grafana provides centralized dashboards for:

Dashboards provide real-time operational visibility.

Each microservice exposes Prometheus metrics.

- Vote Service
- Result Service

Metrics are discovered automatically using ServiceMonitors.

## Kubernetes Monitoring

Infrastructure monitoring includes:

- Node health
- Pod health
- Deployment status
- StatefulSets
- Persistent Volumes
- Cluster resource utilization

This provides complete visibility into cluster operations.

## Security Monitoring

Security events generated by **Falco** are integrated with the observability platform.

Runtime alerts can be forwarded through:

- Alertmanager
- Grafana dashboards
- Slack notifications

This provides centralized visibility into runtime security events alongside infrastructure and application metrics.

## Challenges Encountered

During implementation several monitoring challenges were addressed:

- ServiceMonitor discovery
- Prometheus scrape configuration
- Grafana dashboard customization
- Exporter configuration
- Alert rule tuning
- Namespace monitoring
- Metric labeling

These challenges were resolved through Prometheus target validation, Kubernetes diagnostics, and Grafana dashboard tuning.

## Summary

The platform implements a centralized observability architecture using **Prometheus**, **Grafana**, and **Alertmanager** to provide comprehensive visibility across infrastructure, Kubernetes, applications, databases, and runtime security.
